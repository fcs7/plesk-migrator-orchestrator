# Plesk Migrator Orchestrator

Orquestrador Python: cPanel → Plesk Obsidian.

## Requisitos
- Python 3.8+ (`from __future__ import annotations`, PEP 604, `collections.abc`).
- PyYAML no mesmo Python: `python3.8 -m pip install pyyaml`.
- `run.sh` probe `python3.{12,11,10,9,8}` → `python3` (CentOS/RHEL/Alma 8: `python3` = 3.6).
- Root no Plesk destino.

## Endpoints
- **Plesk dest (magical-liskov):** `ssh -p 2222 root@191.7.26.24` (key auth). Hostname codifica IP: `<id>.<ip-com-traços>.plesk.page` (ex.: `magical-liskov.191-7-26-24.plesk.page`). `systemctl status sshd` revela porta quando SSH timeout.
- **cPanel origem:** 191.7.26.26 (creds em `/etc/plesk-migration.yaml`).

## Logs
- **Orchestrator log_dir:** `/var/log/plesk-migration-orchestrator/` (separado de Plesk Migrator nativo). Per-fase: `copy-{web,mail,db}.log`, `fix-{docroot,limits,mailpath,mail-quota}.log`, `retransfer-attempt-N.log`, `transfer-accounts.log`, `orchestrator.log`, CSVs (`ftp-renames.csv`, `docroot-diff-<dom>.csv`, etc).
- **Plesk Migrator nativo:** `/usr/local/psa/var/modules/panel-migrator/sessions/<session>/` — `debug.log`, `test_all_report.*`, `successful-subscriptions.*`, `accounts_report_tree.*`.
- **SQL inspeção:** `plesk db -Nse "<SQL>"` (sem header). Tabelas chave: `domains`, `hosting` (FK `dom_id`, coluna `www_root` armazena caminho ABSOLUTO), `sys_users`, `mail`, `accounts`, `dns_recs` (FK `dom_id`, não `domain_id`).
- **Tabela `domains` (psa db) — colunas críticas**: `id`, `name`, `cl_id` (FK `clients.id`; 0=admin), `webspace_id` (0=top-level subscription, !=0=filho da subscription), `parentDomainId` (camelCase — não `parent_id`! 0=top-level, !=0=subdomain), `dns_zone_id`. **Subdomains migrados cPanel** ficam em `parentDomainId != 0` E `webspace_id = <parent.id>`. Para listar subdomains de uma subscription: `SELECT name FROM domains WHERE parentDomainId IN (SELECT id FROM domains WHERE name='<parent>')`.

## Comandos
- Wizard (gera YAML interativo, chmod 600, getpass na senha): `sudo ./run.sh --init` (ou `sudo ./run.sh` sem args dispara wizard se `/etc/plesk-migration.yaml` ausente). Override de path: `--config-out /tmp/teste.yaml`.
- Dry-run: `sudo ./run.sh --config /etc/plesk-migration.yaml --dry-run --skip-install`
- Real: `sudo ./run.sh --config /etc/plesk-migration.yaml --skip-install`
- Retomar sessão: `sudo ./run.sh --config /etc/plesk-migration.yaml --skip-install --resume`
- Retomar de step nativo: `sudo ./run.sh ... --resume --only-phase transfer --start-from <step>`
- Validar: `python3.8 -m py_compile plesk_migrator_orchestrator.py`
- Validar (dev local sem python3.8): `python3 -m py_compile plesk_migrator_orchestrator.py`
- Tests: `python3 -m unittest discover -s tests -v` (suite atual ~50 testes; deve ficar verde antes de commit; pytest não está instalado no dev local).

## Armadilhas
- **Conta cPanel órfã** (owner = reseller ausente em `whmapi1 listresellers`): shallow-dump retorna `<resellers/><clients/><domains/>` sem erro. Fix: `whmapi1 modifyacct user=X OWNER=root` (uppercase! `newowner=` ignorado silencioso).
- **rsync exit 23 + "dump.xml: No such file"** no preflight: pmm_agent recebeu seleção vazia, saiu com help. Migration-list só plans, sem domínios.
- **`max_allowed_packet` destino < origem**: preflight bloqueia. Zero-downtime: `SET GLOBAL max_allowed_packet=...` + drop-in `/etc/my.cnf.d/zz-plesk-migration.cnf`.
- `--skip-infrastructure-checks`: pula bloqueio, quebra copy-db. Evitar.
- Retry limpo: `rm -rf /usr/local/psa/var/modules/panel-migrator/sessions/migration-session`.
- **`--resume` valida fingerprint SHA-256 do YAML** (salvo em `<session>/.orchestrator-fingerprint` na 1ª execução). Mismatch = aborta com instrução de fix. Restaure YAML original OU descarte sessão.
- **`--resume` ∧ `--force-regenerate` = mutex** (erro em main + __init__).
- **`--start-from` é raw para plesk-migrator**: steps variam por versão. `plesk-migrator transfer-accounts --help` lista válidos. Comuns: `copy-database`, `copy-web-content`, `copy-mail-content`, `deploy-database`, `deploy-domain`, `restore-hosting`.
- **`--start-from` sem `--only-phase transfer`** rejeitado em main (válido só com transfer/all/default).
- **Backup automático em resume**: `migration-list.pre-resume.<UTC-ts>.bak` antes de skip.
- **Idempotência de transfer-accounts**: plesk-migrator nativo rastreia task-id via pmmcli. Re-rodar sem `--start-from` skipa etapas concluídas (conflict-resolve já validado em execução prévia).
- **Docroot vazio pós-migração**: plesk-migrator pode depositar conteúdo em qualquer subpasta cPanel-style (`public_html/`, `www/`, `web/`) enquanto Plesk default `www-root` = `httpdocs/`. Fase `fix-docroot` (após `copy-db`) escaneia `DOCROOT_CANDIDATES = ("httpdocs", "public_html", "www", "web")` em `/var/www/vhosts/<dom>/`, gera manifest stat-only via `_dir_manifest` (count + bytes + MD5 de `relpath:size` ordenado, não lê conteúdo — rápido em GB), `_pick_docroot` escolhe o candidato não-canonical de maior `total_bytes`. Skip se: httpdocs já populado, tudo vazio, ou hash do escolhido = hash do httpdocs (symlink/hardlink). Aplica `plesk bin subscription -u <dom> -www-root <path>`. Domínios carregados de `<session>/successful-subscriptions.*` (fallback: JSON status/report). Skip-flag: `--skip-fix-docroot`. Isolado: `--only-phase fix-docroot --resume`. Log per-domínio em `<log_dir>/fix-docroot.log` (todos os candidatos com count/bytes/hash truncado).
- **fix-docroot cobre subdomains automaticamente** — após o loop parents, fase roda segundo loop sobre `_load_migrated_subdomains()` (SQL: `SELECT p.name, d.name FROM domains d JOIN domains p ON d.parentDomainId=p.id WHERE d.parentDomainId != 0 AND p.name IN (<parents migrados>)`). Escaneia `SUBDOMAIN_DOCROOT_CANDIDATES` (`<sub_full>`, `<sub_label>`, `<sub_label>/public_html`, `<sub_label>/httpdocs`, `public_html/<sub_label>`, `<sub_full>/httpdocs`, `<sub_full>/public_html`), `_pick_subdomain_docroot` escolhe maior por bytes (skipa quando canonical `<sub_full>` é o melhor OU quando best é duplicata hash do canonical). Aplica `plesk bin subdomain -u <sub-label> -webspace-name <parent> -www-root <relative>` (relative — mesmo trap do parent). Skip-flag dedicado `--skip-fix-docroot-subdomains` (parents continuam sendo processados). Idempotente. Logs em `<log_dir>/fix-docroot.log` com prefixo `[sub]`.
- **`_dir_manifest` retorna 4-tuple** `(count, total, digest, body)`: `body` raw (`"\n".join(sorted entries)`) exposto pra debug e recomparação. Chamadores antigos com 3-tuple unpack (`a, b, c = _dir_manifest(...)`) levantam `ValueError: too many values to unpack`. Mesmo formato em `_remote_dir_manifest`.
- **Locale-aware sort no remoto**: `find ... | sort -u` em cPanel host respeita `LC_COLLATE`. Em sessão SSH com `LANG=pt_BR.UTF-8` herdado, ordem difere de `sorted()` Python (byte-order) → hash DIVERGE espúrio em paths com acento. Sites ASCII são seguros. Preventivo: prefixar `LC_ALL=C` em `_remote_dir_manifest.remote_cmd` se operadores virem DIVERGE em paths não-ASCII.
- **Mail vazio pós-migração**: plesk-migrator pode depositar Maildirs em path cPanel-style (`/var/www/vhosts/<dom>/mail/<dom>/<user>/` ou `/var/qmail/mailnames/<dom>/<user>/` sem subpasta `Maildir/`) enquanto Plesk lê `/var/qmail/mailnames/<dom>/<user>/Maildir/{cur,new,tmp}`. Fase `fix-mailpath` (entre copy-mail e check-mail-passwords) audita read-only e grava `<log_dir>/fix-mailpath.log` com mismatches. NÃO move arquivos (correção manual: rsync + `plesk repair mail -domain <dom>`). Lista contas via `plesk bin mail --list -domain <dom>` (cache por sessão). Skip: `--skip-fix-mailpath`.
- **Senhas de e-mail vazias** (psa.accounts.password NULL): hashes cPanel não-compatíveis com armazenamento Plesk → login IMAP/SMTP falha. Fase `check-mail-passwords` audita via `plesk db -Nse <SQL>` join mail+domains+accounts. Reset opcional via `--reset-mail-passwords`: gera `secrets.token_urlsafe(16)` por conta, aplica `plesk bin mail -u <addr> -passwd <pwd>`, grava `<log_dir>/mail-password-reset.csv` (chmod 600, header `timestamp,email,new_password`). Senhas adicionadas a `sensitive_values` para mascarar em logs. Distribuir via canal seguro fora-de-banda. Skip: `--skip-check-mail-passwords`.
- **Quota mailbox=0 da subscription** (`available: 0` em transfer-accounts): Plesk Migrator cria subscription unplanned com `Limits.mbox_quota=0`. Sem CLI estável (`-mailboxes` rejeitado em unplanned). Fase `fix-limits` (entre transfer e copy-web) faz `UPDATE Limits SET value='-1' WHERE limit_name IN (mbox_quota, max_box, max_subdom, max_db, ...)` por domínio migrado + `plesk repair db -y` pra invalidar cache. Idempotente. Skip: `--skip-fix-limits`.
- **Subscriptions falhando em transfer-accounts** (mailboxes não criadas, sites bloqueados): fase `retransfer-failed` (após fix-limits) auto-reroda `plesk-migrator transfer-accounts --migration-list-file failed-subscriptions.<ts>` até zero falhas, `MAX_RETRANSFER_ATTEMPTS=3` (override `--max-retransfer-attempts N` ou `migration.max_retransfer_attempts` YAML), ou mesma lista 2x consecutivas (raise PhaseExecutionError "progresso estagnado"). Skip: `--skip-retransfer-failed`.
- **Subdomains reservados Plesk** (webmail/mail/ftp/smtp/imap/pop/ns1/ns2 etc): Plesk usa `webmail.<dom>` para webmail nativo; cliente cPanel que tinha esse subdomain bate erro `Incorrect subdomain name: used for accessing webmail`. Fase `sanitize-list` (após `generate-migration-list`) detecta e relata em `<log_dir>/reserved-subdomains-report.csv`. Opt-in `--rename-reserved-subdomains`: reescreve migration-list trocando label conforme `DEFAULT_RESERVED_RENAME` (webmail→correio, mail→email, etc) — override via `migration.reserved_renames` YAML. Backup `migration-list.pre-sanitize.<ts>.bak`. Risco: cliente pode ter URLs hardcoded — por isso opt-in. Skip: `--skip-sanitize-list`.
- **DNS conflicts cPanel-only**: cPanel cria automaticamente registros `cpcontacts.<dom>`, `cpanel.<dom>`, `whm.<dom>`, `webdisk.<dom>` que viram conflito em Plesk (erro `DNS record already exists`). Fase `fix-dns-conflicts` consulta `dns_recs` via `plesk db`, lista hits. Opt-in `--apply-dns-cleanup`: `DELETE FROM dns_recs ... WHERE host REGEXP '^(cpcontacts|cpanel|whm|webdisk)\.'`. Skip: `--skip-fix-dns-conflicts`.
- **Mailbox individual quota=0** (após criação): Plesk-migrator preserva quota cPanel literal (`0` = ilimitado lá, = bloqueio aqui). Conta criada não recebe nada. Fase `fix-mail-quota` (após `check-mail-passwords`) faz `UPDATE mail SET mbox_quota=-1 WHERE dom_id IN (...) AND mbox_quota=0`. Idempotente. Skip: `--skip-fix-mail-quota`.
- **FTP users renomeados silenciosamente** (`user@dom` → `user_dom`): Plesk rejeita `@` em login → renomeia. Cliente não sabe; apps com FTP hardcoded quebram. Fase `fix-ftp-renames` (audit-only) parsea `accounts_report_tree.*` mais recente, regex `Login of FTP user 'X' does not conform to Plesk rules. It was changed to 'Y'`, grava `<log_dir>/ftp-renames.csv` (header `timestamp,original_login,new_login,domain`). Dedup por (original, new). Skip: `--skip-fix-ftp-renames`.
- **fix-mailpath apply mode**: além de auditar (default), `--apply-mailpath-fix` rsync `-a` do path alternativo populado para canonical `/var/qmail/mailnames/<dom>/<user>/Maildir/` + `plesk repair mail -domain <dom> -y`. Não remove origem (operador inspeciona antes). Idempotente (skipa se canonical já populado).
- **Helper `_run_plesk_db(sql, *, fetch=False)`**: executa `plesk db -Nse`. Em dry_run loga e retorna ''. Raise `PhaseExecutionError` em rc≠0. Use pra `UPDATE`/`DELETE` em todas as fases SQL. `_sql_escape()` escapa quote única (suporte minimal — só pra nomes de domínio/limit_name).
- **PHASES_ORDER definitivo**: sanity-check → install → config → list → sanitize-list → filter → preflight → transfer → fix-limits → retransfer-failed → fix-owner → copy-web → copy-mail → fix-mailpath → check-mail-passwords → fix-ftp-passwords → fix-mail-quota → fix-ftp-renames → fix-dns-conflicts → copy-db → fix-docroot → fix-htaccess → test → cleanup-config.
- **Nova flag bool no YAML**: ao adicionar `apply_*` ou `skip.*`, incluir na tupla correspondente em `_validate_config` (~linhas 364/373) para rejeitar `"false"` string (truthy em Python). Documentar também em `config.example.yaml`.
- **Domínios caem em Admin** (subscription owner=root, Customer cPanel não clonado): plesk-migrator nativo pode pular criação do Customer quando cPanel tinha account owner=root (reseller ausente) OU quando `generate-migration-list` emite domínios sob bloco "Admin subscriptions" em vez de `Customer:`. Fase `fix-owner` (entre `retransfer-failed` e `copy-web`) parseia `successful-subscriptions.<ts>` mais recente, extrai mapeamento `Customer: <nome>` → `[domínios]`. Query `psa db`: `SELECT d.name, COALESCE(c.login,'__admin__') FROM domains d LEFT JOIN clients c ON d.cl_id=c.id WHERE d.name IN (...)`. Audit default grava `<log_dir>/owner-mismatches.csv` (`timestamp,domain,expected_customer,proposed_login,current_owner`). Opt-in `--apply-owner-fix`: para cada Customer mismatched cria login slugificado (`_slugify_login`: lowercase `[a-z0-9_]`, começa com letra, max 32) via `plesk bin customer --create <login> -name <nome> -passwd <urlsafe(16)> -email <login>@example.invalid` (skip se `--info` retorna rc=0), depois reassigna via `plesk bin subscription --change-owner <dom> -owner <login>`. Senhas em `<log_dir>/owner-fix.csv` chmod 600 + `sensitive_values` (mascarado em logs). Email é placeholder `@example.invalid` — operador atualiza pós-fix. Domínios sob `Reseller:`/`Plan:`/Admin ficam intactos (sem Customer mapeado). Skip: `--skip-fix-owner`.
- **Webmail subdomain no backup cPanel**: `webmail.<dom>` pode existir no backup cPanel sem aparecer no migration-list. Plesk bloqueia criação (reservado para webmail nativo). `retransfer_failed` detecta "domínio existe em psa.domains + único erro = sites not created on target panel: 'webmail.<dom>'" → aceita como partial success (log warning, não raise). Plesk serve webmail via URL nativa automaticamente. Mesma lógica aplicada na exaustão de `max_attempts`. Subdomain reservado define-se por `RESERVED_PLESK_SUBDOMAINS` (webmail/mail/ftp/ns1/ns2/...). **Classificação per-domain**: `_subscription_only_reserved_failures(domain, session)` filtra hosts por `host == domain or host.endswith("." + domain)` antes do `issubset` — sem isso, falha não-reservada de outra subscription (ex.: `shop.B.com`) no mesmo `accounts_report_tree.*` poluía a classificação de A.com e fazia `retransfer_failed` raise `PhaseExecutionError` em batches mistos. Helper raw `_parse_reserved_subdomain_failure_hosts(session)` expõe a lista de hosts pré-split para reuso.
- **fix-docroot validação cross-server**: após `plesk bin subscription -www-root`, `_validate_docroot_match` SSH no cPanel (`sshpass -e` + env `SSHPASS`, password NUNCA em argv — protege `/proc/<pid>/cmdline` em hosts com shell acesso), roda `LC_ALL=C find ... -printf '%P:%s\n' | LC_ALL=C sort -u` em `/home/<dom-first-label>/public_html` e compara hash com `_dir_manifest` local do diretório escolhido. **`LC_ALL=C` em find E sort é crítico** — sem ele a colação do locale (pt_BR.UTF-8 etc) reordena nomes com acento/special chars de forma diferente do Python `entries.sort()` (bytewise/codepoint) → DIVERGE falso. Pipeline shell guarda body vazio (`if [ -z "$body" ]; then md5=; fi`) — sem isso `md5sum < /dev/null` retornaria `d41d8cd9...` truthy e quebraria o branch `pulada`. `_dir_manifest` chama `entries.sort()` antes do MD5 para casar com ordem `LC_ALL=C find | LC_ALL=C sort -u` (sem isso `os.walk` produz "root-files-first" e hashes divergem em qualquer árvore WP/Joomla). Ambas funções retornam tupla `(count, total, digest, body)` — body é raw `"\n".join("relpath:size")` sem trailing `\n`, usado pelo set-diff em divergência. Stdout SSH separa header de body com sentinel `---BODY---` (parser faz `split("\n---BODY---\n", 1)`). Match → INFO `hash OK`. Diverge → WARNING `hash DIVERGE src(N1/B1/h1) dst(N2/B2/h2) — N só no src, M só no dst — ver docroot-diff-<dom>.csv` E grava `<log_dir>/docroot-diff-<domain>.csv` (header `timestamp,side,path,size`, side ∈ {src,dst}) com `set(src_body.splitlines()) ^ set(dst_body.splitlines())` listando exatamente quais arquivos sumiram (`side=src` = só no cPanel) ou apareceram (`side=dst` = só no Plesk). OSError gravando CSV é log+swallow (best-effort, não aborta). Falha SSH (sshpass ausente, conexão recusada, path inexistente) → WARNING `validação cross-server pulada`. Nunca aborta. Username cPanel resolvido via `_resolve_cpanel_user` (SSH `grep /etc/userdomains` cacheado por sessão); fallback heurístico `dom.split('.', 1)[0]` apenas se lookup falhar (sshpass ausente, host inacessível, domínio fora do `/etc/userdomains`).
- **`plesk bin subscription -www-root <path>` é RELATIVO ao vhost root** (`subscription --help`: "relative to the subscription root"). Passar absoluto (ex.: `/var/www/vhosts/dom/public_html`) → Plesk concatena vhost-root + arg → `/var/www/vhosts/dom/var/www/vhosts/dom/public_html` (inexistente, Apache 403 em tudo). Sempre passar só `public_html`/`httpdocs`/`www`. Validar pós-aplicação via `plesk db -Nse "SELECT name,www_root FROM domains d JOIN hosting h ON d.id=h.dom_id"`.
- **`/etc/userdomains` cPanel = mapping canônico** `<domain>: <user>` (1:1, mantido pelo WHM). Fonte oficial pra resolver user real — heurística `domain.split(".",1)[0]` quebra com renames cPanel-style (`opiniao` → `opiniaoi`). `_validate_docroot_match` usa SSH `grep "^<dom>:" /etc/userdomains` com cache de sessão; falha de lookup degrada para first-label (back-compat).
- **WHM/cPanel password hashes não funcionam em Plesk para FTP nem mail**. cPanel usa crypt MD5; Plesk Linux espera SHA-512. Sub-FTP users (`user@dom` → `user_dom`) batem `530 Login incorrect` mesmo com path/perm OK. `check-mail-passwords` reseta mail; `fix-ftp-passwords` (opt-in `--reset-ftp-passwords`) reseta FTP via `plesk bin ftpsubaccount --update <login> -passwd "" -domain <dom>` com senha via env `PSA_PASSWORD` (NÃO em argv — evita leak `/proc/<pid>/cmdline`, mesmo padrão sshpass `-e`). **`plesk bin ftpuser` NÃO existe** em Plesk Obsidian 18 — `ftpsubaccount` é o CLI canônico para Plesk additional FTP accounts (categoria à qual pertencem os `user@dom` cPanel renomeados). CSV chmod 600 em `<log_dir>/ftp-password-reset.csv`. **Fonte autoritativa = `accounts_report_tree.*` da sessão (regex `Login of FTP user 'X' does not conform...`)** via helper `_parse_ftp_renames_from_report` — NÃO SQL. Subscription main FTP user (criado novo pelo Plesk, hash OK nativo) é intencionalmente NÃO tocado; pra reset manual use `plesk bin subscription --update <dom> -passwd` (também rotaciona SSH/painel — fora de escopo). Independente da ordem PHASES_ORDER vs fix-ftp-renames porque ambos parseiam o mesmo report.
- **Reserved-subdomain classifier case-insensitive**: `_subscription_only_reserved_failures` faz `domain.lower()` e `h.lower()` antes de comparar em `_parse_reserved_subdomain_failure_hosts`. Sem isso hosts cPanel-style mixed-case (`Webmail.OPINIAO.INF.BR`) caíam fora do filtro per-domain → falsas falhas raise em retransfer_failed.
- **Padrão CSV de diagnóstico**: fases que afetam múltiplos domínios gravam `<log_dir>/<feature>-<dom>.csv` (per-domínio) ou `<log_dir>/<feature>.csv` (agregado), header iniciando com `timestamp,...`. Vivos: `ftp-renames.csv`, `owner-fix.csv` (chmod 600 — senhas), `owner-mismatches.csv`, `mail-password-reset.csv` (chmod 600), `reserved-subdomains-report.csv`, `docroot-diff-<dom>.csv`. OSError gravando CSV é log+swallow (best-effort, nunca aborta pipeline). Sensíveis vão em `sensitive_values` para mascarar em logs.
- **Manifest 4-tupla `(count, total, digest, body)`**: helpers `_dir_manifest` (local stat-only) e `_remote_dir_manifest` (SSH cPanel via `sshpass -e`) retornam tupla 4-elementos. `body` é raw `"\n".join("relpath:size")` sem trailing `\n`; hash MD5 desse body. **Ordenação `LC_ALL=C`** no shell remoto (ambos `find` e `sort -u`) casa com `entries.sort()` bytewise/codepoint do Python — sem isso locale pt_BR.UTF-8 diverge em nomes com acento. Transport SSH separa header (`COUNT/TOTAL/MD5`) de body com sentinel `---BODY---`; parser Python faz `stdout.split("\n---BODY---\n", 1)`. `UnicodeDecodeError` em subprocess.run com `text=True` (cPanel com filenames Latin-1) é tratado igual a SSH failure → retorna `(0, 0, "", "")`. Padrão reutilizável pra qualquer fase futura que precise comparar árvore cross-server (copy-mail, copy-db).
- **`test-all` é advisory (não-bloqueante)** — chamado direto via `subprocess.run(check=False)`. Após execução, `_classify_test_all_report` lê o `test_all_report.<ts>` mais recente (preferindo sufixo timestamp sobre `.json`) e classifica em `ok`/`soft`/`hard`. **Patterns `_TEST_ALL_ADVISORY_PATTERNS`**: `The HTTP status code of a web page has changed`, `The hosting checker cannot connect`, `530 Login incorrect`, `User password is encrypted`, `Cannot verify access`/`password`. Verdict `soft` (todos os erros advisory) → log WARNING + segue. `hard` (erro não-mapeado, signal exit, report ausente quando rc≠0) → raise `PhaseExecutionError`. `ok` (0 erros) → INFO. Resumo gravado em `<log_dir>/test-all-summary.txt`. Caso especial: rc=0 sem report (plesk-migrator idle) é tratado como `ok`. **Não use `_run` para test-all** — `_run` tem `check=True` por default e abortaria o pipeline em qualquer rc≠0.
- **fix-htaccess** — fase opt-in-apply (audit por default) que detecta diretivas cPanel-only em `.htaccess` migrados via `rglob` no vhost root de cada parent. **Padrões em `_HTACCESS_PROBLEM_PATTERNS`**: `suPHP_*` (cPanel-only, Apache 500 `Invalid command 'suPHP_ConfigPath'`), `AuthUserFile`/`AuthGroupFile` apontando para `/home/<user>/...` (path inexistente em Plesk), `php_value`/`php_admin_value` (Plesk PHP-FPM ignora), `Action ... /cgi-sys/`, `AddHandler cpanelphp|cgi-script`, `FCGIWrapper`. Audit grava `<log_dir>/htaccess-audit.csv` (header `timestamp,domain,path,line_number,category,snippet`). `--apply-htaccess-fix` comenta linha com prefixo `#FIX# ` + backup `.htaccess.pre-htaccess-fix-<ts>.bak`. **Idempotente** (linhas já com `#FIX# ` não são re-processadas; backup só é criado se algo mudar). Posicionada entre `fix-docroot` e `test` em `PHASES_ORDER`. Skip-flag `--skip-fix-htaccess`.
